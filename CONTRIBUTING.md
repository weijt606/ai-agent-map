AI Agent Map is not trying to become the longest list of agent links.

The contribution bar is narrower on purpose: if a page does not help readers make a clearer decision, it probably does not belong here.

## What To Contribute

- New agent profiles that regularly appear in real-world shortlists.
- Use-case guides that help readers choose from a concrete problem.
- Comparison pages where the projects genuinely compete for the same decision.
- Accuracy fixes when a product has changed, been renamed, or been misunderstood.

## What Not To Contribute

- Raw link dumps.
- Marketing copy copied from vendor pages.
- Unverified naming or product boundaries presented as facts.
- Generic rankings with no decision value.
- Vendor logos, screenshots, or brand assets unless you clearly have the right to use them.

## Core Writing Rules

- Write for selection, not hype.
- Always state both fit and anti-fit.
- Prefer practical trade-offs over architecture buzzwords.
- Keep claims scoped. If something is an editorial judgment, write it that way.
- If the public boundary of a product is messy, say so explicitly.

## Verification Standard

Before adding or changing a profile, verify against current primary sources such as:

- Official product pages
- Official docs
- Official repositories
- Official release notes or changelogs

When a project has multiple surfaces, do not collapse them into one vague description. Examples:

- Product vs. CLI companion
- Managed product vs. open-source SDK
- Current product vs. deprecated API

If a mapping is still uncertain, mark it as provisional in the page and in the index.

## Legal And Copyright Guardrails

- Do not paste vendor marketing paragraphs.
- Do not reuse proprietary screenshots by default.
- Do not use vendor logos in banners or comparison art by default.
- Prefer original diagrams, original illustrations, and plain text labels.
- Use trademarked product names only to identify the product, not to imply endorsement or partnership.

## Recommended Contribution Flow

1. Pick a page type: profile, use case, or comparison.
2. Verify the current public product boundary first.
3. Write the shortest version that still helps a reader decide.
4. Add caveats where naming, maturity, or positioning is ambiguous.
5. Check neighboring indexes and comparison tables so wording stays consistent.

## Profile Checklist

- Is the official entry point current?
- Is the product described as it exists now, not as it existed in an older launch post?
- Does the page separate product surfaces when they differ materially?
- Does the page clearly say when to pick it?
- Does the page clearly say when not to pick it?
- Are any provisional mappings visibly marked?

## Style Notes

- English-default root docs keep Chinese mirrors under `zh/`.
- Keep tone practical, light, and readable.
- Avoid academic or overly abstract phrasing.
- Prefer original wording even when summarizing official material.

## If You Are Unsure

Open the contribution as a narrow correction instead of a broad rewrite.

Accuracy beats coverage.
