一句话：Aider 适合想把 AI 结对编程留在终端里、同时又不丢 git 节奏的人。

## 一眼判断

| 项目 | 结论 |
| --- | --- |
| 厂商 | Aider-AI |
| 路线 | 终端优先的 AI 结对编程 agent |
| 是否开源 | 是 |
| 最适合 | 以仓库为中心、强调 git review loop 的本地编码工作流 |
| 主要代价 | API key、模型选择和终端使用习惯都是 operating model 的一部分 |
| 官网 | https://aider.chat/ |
| GitHub 仓库 | https://github.com/Aider-AI/aider |

## 什么时候选它

- 你本来就长期工作在终端里，不想为了 AI 换一整个 IDE。
- 你希望 diff、commit 和 git discipline 仍然是 agent 改代码的主轴。
- 你希望模型选择足够开放，包含多家 hosted 模型和本地模型。

## 什么时候别选

- 你想要更 polished 的团队 UI 或 manager-friendly 的云端看板。
- 你把完全脱手的后台委派视为主工作流。
- 你真正需要的是更宽的 multi-agent platform，而不是终端里的 pair programmer。

## 能力侧写

| 维度 | 判断 | 备注 |
| --- | --- | --- |
| Terminal workflow | 很强 | 终端就是天然主场 |
| Git integration | 很强 | 这是它最明显的差异点 |
| Model flexibility | 强 | 支持多种 hosted 和本地模型 |
| IDE adjacency | 中 | 可以配合，但不是主 operating model |
| Explicit approval | 中 | 控制更多来自 diff 和 git，而不是每一步确认 |

## 使用代价

复杂度是 Medium。安装不算重，但最终效果很依赖模型选择、API 配置，以及团队是否真接受终端优先的工作方式。

## 最后一句

Aider 是“我想要 AI 结对编程，但不想离开终端，也不想放弃 git discipline”这条路线里很有代表性的开源答案。
